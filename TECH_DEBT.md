| TD-010 | **The visual gate was asserting that components look like their *broken* selves.** Two compounding faults: (1) the budget was a ratio of the **full page** — 1280×720 = 921.600 px, so `maxDiffPixelRatio: 0.01` allowed **9.216 differing px** against a Checkbox showcase of **1.024 px**, i.e. blind to any small component; (2) worse, `toHaveScreenshot` polls and stops the moment a frame **matches** the expected image, and the harness's CSS is JS-injected by the Vite dev server (TD-011) — so a baseline recorded in the unstyled window matched an early unstyled frame on every later run, forever. **Fixed** by an absolute budget (`maxDiffPixels: 100`, measured: two `--update-snapshots` runs are byte-identical across all 60, so there is no jitter band) + regenerating baselines on CI Linux, which changed **41 of 60** — `Callout` by 554.392 px (its soft backgrounds were absent), `AlertBanner` by 335.111. Proof of the blindness: #69 changed the Checkbox's border *and* added a fill, and `visual` reported SUCCESS. Together with TD-002 and TD-003, this is the full answer to how an invisible checkbox shipped and survived. | 2026-07-15 (PR #72) |
| TD-004 | (Was: harness without `@tailwindcss/vite` → colors fall back to browser defaults.) **Resolved by migrating colored arbitrary values** `*-[var(--ui-*)]` (1482 across 104 components, variant-aware) to static classes in `src/ui-classes.css`. Components now get colors from plain CSS imported per-file — independent of any consumer/harness Tailwind scan (also fixes the "invisible component" bug class + unblocks contrast testing). | 2026-06-25 (PR `ds/css-classes-and-lint`) |
